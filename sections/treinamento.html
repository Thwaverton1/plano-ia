🔬 Trilha — Treinamento de Modelos de IA
Do zero ao ML engineer capaz de treinar, ajustar e escalar modelos de grande porte · ~18 meses · A espinha dorsal de qualquer sistema de IA sério
Esta trilha é sobre dominar o processo completo de criar modelos: desde coleta de dados até deploy em produção, passando por arquitetura, treinamento eficiente, avaliação rigorosa e otimização. É o que as big techs mais buscam.

Onde essa especialização tem mais impacto

Papéis profissionais que dependem desta trilha

  • ML Engineer: constrói e mantém pipelines de treinamento em produção
  • Research Engineer: implementa papers de pesquisa em código eficiente
  • Foundation Model Engineer: treina LLMs/VLMs de grande escala
  • MLOps Engineer: garante que modelos funcionem em produção com qualidade
  • AI Engineer: adapta modelos existentes para problemas específicos

Habilidades que definem o especialista

  • Entende por que o treinamento falha e como corrigir
  • Otimiza uso de GPU/TPU para máxima eficiência
  • Escala treinamento de 1 GPU para centenas
  • Avalia modelos corretamente — além de accuracy
  • Detecta e corrige problemas em dados antes de treinar
  • Documenta experimentos de forma reproduzível

Fase 1 — Dominar o loop de treinamento (Meses 1–3)

Mês 1 — PyTorch profundo
  • Autograd: como gradientes fluem pelo grafo computacional
  • custom Dataset, DataLoader, collate_fn
  • Training loop do zero: forward, loss, backward, optimizer step
  • Checkpointing: salvar e restaurar estado completo do treinamento
  • Profiling: torch.profiler para identificar gargalos
  • Projeto: treinar classificador completo com checkpointing e métricas
Mês 2 — Dados: a parte mais importante
  • Data quality: detectar e remover dados ruins (ruído, duplicatas, leakage)
  • Data imbalance: oversampling, undersampling, class weights, focal loss
  • Data augmentation avançada: Mixup, CutMix, AugMix
  • Datasets eficientes: WebDataset, Arrow/Parquet para dados grandes
  • Label quality: Cleanlab para detecção de erros de anotação
80% dos problemas de modelos ruins são problemas de dados ruins. Esta semana é mais valiosa do que parece.
Mês 3 — Otimizadores e learning rate
  • SGD vs Adam vs AdamW — quando usar cada um e por quê
  • Learning rate schedules: cosine annealing, warmup, cyclic LR
  • Gradient clipping: estabilizar treinamento de LLMs
  • Weight decay e regularização L2: relação com AdamW
  • Hyperparameter tuning: Optuna para busca eficiente
  • Projeto: ablation study sistemático comparando otimizadores e LR schedules

Fase 2 — Treinamento eficiente em GPU (Meses 4–7)

Mês 4 — Precisão mista e memória
  • FP16 / BF16: treinar em meia precisão sem perder qualidade
  • torch.cuda.amp: automatic mixed precision
  • Gradient accumulation: simular batch grande com GPU pequena
  • Ativação de checkpointing: trocar memória por recomputação
  • Flash Attention: atenção eficiente em memória para Transformers
Mês 5 — Fine-tuning eficiente de LLMs
  • LoRA: adapters de baixo ranque — fine-tuning sem reescrever pesos
  • QLoRA: LoRA com quantização 4-bit para rodar em GPUs modestas
  • Unsloth: aceleração de fine-tuning de LLMs (2-5x mais rápido)
  • Instruction tuning: formato de dados para fine-tuning em instruções
  • Projeto: fine-tuning de Llama 3 / Gemma para tarefa específica com QLoRA
Com QLoRA, é possível fazer fine-tuning de modelos 7B-13B em uma GPU de 16GB. Isso mudou o jogo em 2023.
Mês 6 — Treinamento distribuído
  • Data parallelism: DDP (DistributedDataParallel) em múltiplas GPUs
  • Model parallelism: dividir modelo entre GPUs (tensor e pipeline parallel)
  • DeepSpeed: ZeRO stages 1/2/3 para modelos que não cabem em 1 GPU
  • Accelerate (HuggingFace): abstração para multi-GPU e multi-node
  • Monitorar treinamento distribuído: wandb, MLflow
Mês 7 — Avaliação rigorosa
  • Além de accuracy: calibração, fairness, robustez, distribuição de erro
  • Benchmarks para LLMs: MMLU, HumanEval, GSM8K, MT-Bench
  • Evaluation harness (EleutherAI): avaliar LLMs de forma padronizada
  • Análise de erros: onde o modelo falha sistematicamente
  • Sliced evaluation: performance por subgrupo (classe, idioma, domínio)

Fase 3 — Modelos grandes e produção (Meses 8–13)

Mês 8–9 — Pré-treinamento do zero
  • Preparação de dados de pré-treinamento: deduplicação, filtragem, mixing
  • Tokenização: BPE, WordPiece, SentencePiece — treinar tokenizer próprio
  • nanoGPT (Karpathy): implementar GPT do zero, escalar progressivamente
  • Scaling laws: como prever performance antes de treinar
  • Infraestrutura: SLURM, Kubernetes para jobs de treinamento em cluster
Recurso: "nanoGPT" do Andrej Karpathy no YouTube — implementar isso muda como você enxerga LLMs para sempre
Mês 10 — Otimização para inferência
  • Quantização: GPTQ, AWQ, GGUF — modelos menores sem perder qualidade
  • Pruning: remover parâmetros desnecessários
  • Knowledge distillation: treinar modelo menor para imitar modelo maior
  • Speculative decoding: acelerar inferência de LLMs
  • vLLM: serving de LLMs com throughput máximo em produção
Mês 11–12 — MLOps para modelos de grande porte
  • Experiment tracking: W&B com sweeps para buscas de hiperparâmetros
  • Model registry: versionar, documentar e promover modelos
  • CI/CD para ML: testar modelos automaticamente antes de promover
  • Monitoramento em produção: drift, performance, latência, custo
  • A/B testing: como testar novos modelos com tráfego real
Mês 13 — Projeto integrador final
  • Treinar modelo completo do zero ou fazer fine-tuning avançado
  • Pipeline: dados → treinamento → avaliação → otimização → deploy → monitoramento
  • Documentação de experimentos reproduzíveis (modelo card)
  • Publicar modelo no HuggingFace Hub
  • Post técnico detalhado descrevendo todo o processo e aprendizados
Ter um modelo no HuggingFace Hub com downloads e README completo é um dos portfólios mais fortes que você pode ter.
Marco da trilha: modelo publicado no HuggingFace Hub com card completo. Pipeline de treinamento reproduzível no GitHub. Experimentos documentados no W&B (link público). Isso te diferencia da esmagadora maioria dos candidatos.
